# RetailPulse Self-Service Automation Plan

## Summary
- **Recommended architecture:** use **Databricks as the backend and system of record**, and use **a Streamlit front end inside Databricks Apps** as the self-service upload and mapping UI.
- **Do not use GitHub Actions as the dataset upload path.** Use GitHub Actions for CI/CD, deployment, smoke tests, and scheduled health checks only.
- **Do not use standalone Streamlit as the primary platform** for v1. It should be a UI layer over Databricks, not a second backend.
- **Production note:** the current Free Edition workspace is not a valid final production target. The real implementation should move to a **paid Databricks workspace** before calling this production, because Free Edition has no SLA/commercial posture and severe limits.
- **Product goal:** support multiple retail datasets in one workspace, with a flexible mapping UI, a canonical retail contract behind the scenes, automatic Databricks processing, parameterized dashboards, and downloadable insight files.

## Important Interfaces
- **New input model:** users upload either:
  - one wide order-item transaction CSV, or
  - a file bundle with `transactions.csv` plus optional `products.csv`, `customers.csv`, `stores.csv`
- **Required minimum grain:** order-item detail. The first automated system will not support arbitrary summary-only retail CSVs.
- **New control artifacts:**
  - `dataset_id` as the primary run key across the whole system
  - `mapping_manifest.json` generated by the UI and stored in a control location
  - `control_dataset_registry` table for status, paths, run ids, and artifact links
  - `control_mapping_manifests` table for normalized mapping metadata
  - `control_run_log` table for job lifecycle and failures
- **New output artifact pack per dataset:**
  - `executive_summary.md`
  - `data_quality_report.json`
  - `kpi_snapshot.csv`
  - `cross_sell_recommendations.csv`
  - `segment_profiles.csv`
  - `demand_windows.csv`
  - `action_recommendations.csv`
  - `experimental_predictive_metrics.csv` only when that lane is enabled
- **Dashboard interface change:** all dashboard queries and report-pack outputs become `dataset_id` aware. The dashboard gets a dataset selector and shows one retail dataset at a time.

## Key Changes
### 1. Platform choice and deployment model
- Keep **Databricks** for:
  - file landing in Unity Catalog volumes
  - Delta tables
  - transformation jobs
  - SQL warehouse
  - AI/BI dashboard
  - output artifact storage
- Use **Databricks Apps + Streamlit** for:
  - file upload
  - file-role assignment
  - column mapping
  - run submission
  - run-status view
  - artifact download links
- Use **GitHub Actions** for:
  - tests
  - bundle validation/deploy
  - app deployment
  - smoke-test execution
  - optional scheduled regression checks
- Explicitly reject these as the primary v1 architecture:
  - GitHub as a user data ingress surface
  - plain external Streamlit as the main backend
  - Free Edition as the final production home

### 2. Introduce a canonical retail data contract
- Replace the current Instacart-only file contract with a **canonical retail model** that the UI maps into.
- Canonical required entities:
  - `transactions`: required
  - `products`: required unless product attributes are embedded in transactions
  - `customers`: optional
  - `stores`: optional
- Canonical required fields for `transactions`:
  - `order_id`
  - `customer_id`
  - `product_id`
  - `quantity`
  - one of:
    - `order_timestamp`, or
    - `order_date` plus `order_hour`
- Canonical useful optional fields:
  - `unit_price`
  - `store_id`
  - `channel`
  - `promotion_flag`
  - `city`
  - `state`
- Derived fields computed by RetailPulse, not required from the source:
  - `order_dow`
  - `order_hour_of_day`
  - `days_since_prior_order`
  - `order_number`
  - `reordered`
  - daypart
- Add hard validation rules in the mapping layer:
  - no run starts without the required canonical fields
  - no duplicate canonical mapping for a required field
  - timestamps must parse successfully above a configured threshold
  - `order_id` + `product_id` uniqueness issues are reported, not silently ignored

### 3. Refactor the pipeline for multi-dataset operation
- Stop using overwrite-only single-dataset semantics for bronze/silver/gold/report tables.
- Introduce `dataset_id` into all normalized bronze, silver, gold, mart, and report tables.
- Process each upload as one isolated dataset slice inside the same shared tables.
- All downstream notebooks must:
  - read only rows for the current `dataset_id`
  - write results tagged with the same `dataset_id`
  - overwrite or merge only that dataset slice, not the full table
- Add a new early pipeline stage:
  - profile uploaded files
  - load mapping manifest
  - normalize raw CSVs into canonical bronze tables
- Keep the existing analytics logic where possible, but generalize it to canonical fields instead of Instacart filenames.
- Preserve the current dashboard/report outputs as the guaranteed v1 output pack.

### 4. Replace the current upload checkpoint with a real ingestion workflow
- Current `01_sample_and_upload.py` only validates pre-uploaded Instacart files. Replace that concept with:
  1. upload to landing volume path `/Volumes/<catalog>/<schema>/<landing_volume>/datasets/<dataset_id>/raw/`
  2. save mapping manifest in `/Volumes/<catalog>/<schema>/<control_volume>/manifests/<dataset_id>.json`
  3. register dataset row in `control_dataset_registry`
  4. trigger processing job
- Recommended trigger model:
  - **v1:** app-triggered job run after mapping is submitted
  - **v2:** file-arrival trigger on the landing path after upload finalization
- Use a status state machine in the registry:
  - `uploaded`
  - `mapped`
  - `validating`
  - `processing`
  - `published`
  - `failed`

### 5. Build a self-service upload and mapping app
- Create one Databricks App using Streamlit with these screens:
  1. `Upload`
  2. `Assign file roles`
  3. `Map columns`
  4. `Validate`
  5. `Run analysis`
  6. `Status and outputs`
- Upload screen:
  - support multiple CSVs
  - support directory upload
  - enforce allowed file types
  - enforce size limits and warn on large files
- Mapping screen:
  - auto-detect likely matches by header names
  - require user confirmation for required fields
  - show sample values before finalizing
- Validation screen:
  - schema fit
  - null-rate checks
  - timestamp parsing checks
  - duplicate-key checks
  - minimum row-count checks
- Outputs screen:
  - dashboard link
  - latest job run link
  - downloadable artifact pack
  - data quality summary
- Keep the app internal. Do not build external customer auth or multi-tenant public access in this phase.

### 6. Automate dashboards and insight files
- Make the existing AI/BI dashboard dataset-aware instead of store-hardcoded.
- Add a `dataset_id` parameter or selector to the canonical SQL query pack.
- Add one `dataset summary` page or widget group that shows:
  - store / dataset name
  - upload date
  - row counts
  - data freshness
- Keep current descriptive outputs:
  - executive overview
  - order behavior
  - recommendations and segments
  - execution and data quality
  - experimental insights and performance
- Add automated **prescriptive outputs** as release-safe rule-based files:
  - `cross_sell_recommendations.csv`
  - `segment_targeting_actions.csv`
  - `demand_window_actions.csv`
  - `inventory_watchlist.csv`
  - `action_recommendations.csv`
- Treat current classifier/regression as optional `Experimental Insights`, not production action drivers.
- Do **not** automate business decisions from the current supervised models.

### 7. Redesign predictive analysis before calling it production-safe
- The current supervised notebooks stay out of the operational path until redesigned.
- Phase the predictive layer this way:
  - **v1 operational output:** descriptive + recommendation + segmentation + rule-based prescriptive files
  - **v2 predictive output:** leakage-safe models only after the canonical retail contract is stable
- When predictive redesign begins:
  - remove `cluster_id` leakage from the classifier path
  - replace time-leaky user aggregates with prior-only features
  - version model inputs and outputs by `dataset_id` and model version
  - write prediction outputs to separate tables and files, not into KPI tables

### 8. GitHub Actions and repo automation
- Extend GitHub Actions into three workflows:
  - `ci.yml`: unit tests, notebook mirror check, schema-contract tests
  - `deploy-platform.yml`: deploy Databricks bundle, queries, dashboards, and app
  - `smoke-test.yml`: run a small fixture dataset end to end and validate artifacts
- Keep GitHub Actions responsible for **code promotion**, not human dataset upload.
- Add contract tests for:
  - mapping manifest parsing
  - canonical schema validation
  - per-dataset isolation
  - artifact generation
  - dashboard query parameter behavior
- Add a fixture dataset in test assets that mimics a generic retail order-item package, not just Instacart naming.

### 9. Documentation and README overhaul
- Rewrite `README.md` so it explains the **future automated system**, not only the current Instacart demo.
- Required README sections:
  - what RetailPulse does now
  - target self-service architecture
  - supported input package
  - required minimum data grain
  - upload-and-run flow
  - dashboard and artifact outputs
  - limitations and unsupported inputs
  - Databricks vs Streamlit architecture decision
  - production note about Free Edition
- Add new docs:
  - `Docs/Automation Architecture.md`
  - `Docs/Canonical Retail Data Contract.md`
  - `Docs/Upload And Mapping Guide.md`
  - `Docs/Artifact Output Spec.md`
  - `Docs/Operations And Recovery.md`
- README should link to those docs and give two user paths:
  - operator / developer
  - business uploader / reviewer

## Test Plan
- **Contract tests**
  - valid single wide CSV maps successfully
  - valid multi-file bundle maps successfully
  - missing required canonical fields blocks the run
  - bad timestamps and duplicate keys are surfaced clearly
- **Pipeline tests**
  - two different `dataset_id`s can coexist in one workspace
  - rerunning one dataset only refreshes that dataset slice
  - dashboard queries return only the selected dataset
  - artifact pack is generated for each successful dataset
- **UI tests**
  - multi-file upload works
  - mapping wizard persists manifest correctly
  - run status updates from upload to published
  - failed runs show actionable validation messages
- **Databricks release tests**
  - bundle validate/deploy succeeds
  - app deploy succeeds
  - file upload to volume succeeds
  - processing job succeeds on a fixture dataset
  - smoke checks pass for required tables and artifacts
- **Acceptance criteria**
  - a new store dataset can be uploaded without code edits
  - the user can map columns through the UI
  - the system publishes a dataset-aware dashboard and output pack
  - multiple datasets coexist in one workspace
  - no production-safe doc claims rely on the current leakage-prone supervised models

## Assumptions
- The first serious implementation target is an **internal B2B/self-service system**, not a public consumer product.
- Multiple retail datasets must coexist in one workspace, so the pipeline will be refactored around `dataset_id`.
- Minimum supported source grain is **order-item detail**; “any retail CSV” is interpreted as “any retail CSV that can be mapped into the canonical order-item contract.”
- **Default recommendation:** Databricks backend + Databricks App with Streamlit UI + GitHub Actions for CI/CD.
- **Production requirement:** move off Databricks Free Edition before calling this production. Official Databricks docs currently position Volumes as the recommended ingestion file store, GitHub Actions as CI/CD automation, Databricks Apps as a serverless app layer that supports Streamlit, and Free Edition as a limited no-SLA environment with one app per account and 24-hour app runtime limits.
